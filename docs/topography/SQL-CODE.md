## Database Role and Architecture

The PEQ (ProjectEQ) database is the single source of truth for all persistent game state. It stores:

- **Game content**: NPCs, items, spells, zones, loot tables, quests, factions, tradeskill recipes
- **Player state**: characters, inventory, skills, spells learned, quest progress, corpses
- **Server configuration**: rule values, spawn conditions, content flags
- **Companion NPCs**: both the "bot" system (player-created party members) and "merc" system (hireable mercenaries)

The C++ server uses **two database connections** per zone process:

| Connection | Variable | Purpose |
|---|---|---|
| `database` | `ZoneDatabase` / `WorldDatabase` | Player state, accounts, operational data |
| `content_db` | Same type | Game content (NPCs, items, spells, zones) -- read-only at runtime |

Both connect to the same MariaDB instance but the separation allows content to potentially be on a separate read-only replica.

**Inheritance chain**: `Database` -> `SharedDatabase` -> `ZoneDatabase` / `WorldDatabase`

Source: `common/database.h`, `common/shareddb.h`, `zone/zonedb.h`, `world/worlddb.h`

## C++ Access Patterns

### Repository Layer (Auto-Generated)

Every table has an auto-generated repository header in `common/repositories/base/base_*_repository.h`, generated by `utils/scripts/generators/repository-generator.pl`. These provide:

- `TableName()` -- returns the SQL table name
- `PrimaryKey()` -- returns the primary key column name
- `Columns()` / `SelectColumns()` -- column lists
- `BaseSelect()` / `BaseInsert()` -- query templates
- CRUD methods: `FindOne()`, `FindAll()`, `InsertOne()`, `UpdateOne()`, `DeleteOne()`, `DeleteWhere()`
- `NewEntity()` -- creates a default-initialized struct

Customization goes in `common/repositories/*_repository.h` (extends the base).

### Direct SQL

Older code paths use `database.QueryDatabase()` with hand-written SQL. Newer code uses the repository pattern. The codebase is gradually migrating to repositories.

### Loading Pattern

At zone boot:
1. `content_db.LoadNPCTypesData(0, true)` -- bulk-loads all NPC types for the zone
2. Spawn2/SpawnGroup/SpawnEntry loaded via `LoadSpawnGroups()`
3. Items, spells loaded into shared memory by `SharedDatabase`

## NPC System

### npc_types (the central content table)

The `npc_types` table is the heart of the content database. Every NPC in the game -- monsters, quest NPCs, pets, merchants, bankers -- is defined here. With ~150 columns, it contains:

**Identity**: `id`, `name`, `lastname`, `level`, `race`, `class`, `gender`, `bodytype`

**Combat stats**: `hp`, `mana`, `mindmg`, `maxdmg`, `AC`, `ATK`, `attack_speed`, `attack_delay`, `attack_count`, `STR/STA/DEX/AGI/INT/WIS/CHA`, `MR/CR/DR/FR/PR/Corrup/PhR`

**Appearance**: `texture`, `helmtexture`, `size`, `face`, `luclin_hair*`, `luclin_eye*`, `luclin_beard*`, `armortint_*`, `d_melee_texture1/2` (weapon models), `light`, `model`, `herosforgemodel`

**Behavior**: `npcspecialattks` (string-encoded special attacks), `special_abilities` (CSV of numbered abilities), `aggroradius`, `assistradius`, `runspeed`, `walkspeed`, `see_invis`, `see_invis_undead`, `see_hide`, `see_improved_hide`, `trackable`, `findable`, `flymode`, `stuck_behavior`, `always_aggro`, `npc_aggro`

**Cross-references** (implicit foreign keys):
- `loottable_id` -> `loottable.id`
- `merchant_id` -> `merchantlist.merchantid`
- `npc_spells_id` -> `npc_spells.id`
- `npc_spells_effects_id` -> `npc_spells_effects.id`
- `npc_faction_id` -> `npc_faction.id`
- `adventure_template_id` -> `adventure_template.id`
- `trap_template` -> trap template id
- `emoteid` -> `npc_emotes.emoteid`
- `armortint_id` / `npc_tint_id` -> `npc_types_tint` lookup

**Charm-specific**: `charm_ac`, `charm_min_dmg`, `charm_max_dmg`, `charm_attack_delay`, `charm_accuracy_rating`, `charm_avoidance_rating`, `charm_atk` -- stats used when the NPC is charmed by a player.

**Content filtering**: `min_expansion`, `max_expansion`, `version` -- controls which expansion an NPC appears in.

**Scaling**: `scalerate`, `maxlevel` -- for auto-scaling NPCs.

**Misc**: `spawn_limit`, `private_corpse`, `unique_spawn_by_name`, `underwater`, `isquest`, `isbot`, `rare_spawn`, `skip_global_loot`, `raid_target`, `keeps_sold_items`, `is_parcel_merchant`, `multiquest_enabled`, `exp_mod`, `faction_amount`, `greed` (merchant greed factor).

**Key C++ access**: `content_db.LoadNPCTypesData(npc_type_id)` loads a single NPC type. Called with `(0, true)` for bulk load at zone boot. Defined in `zone/zonedb.cpp`.

## Spawn System

How NPCs get placed in the world. This is a four-table chain:

```
spawn2  --(spawngroupID)-->  spawngroup  <--(spawngroupID)--  spawnentry  --(npcID)-->  npc_types
                                                                          (with chance %)
```

### spawn2 (spawn point)

Defines a physical location in a zone where something can spawn.

| Column | Purpose |
|---|---|
| `id` | Unique spawn point ID |
| `spawngroupID` | FK to `spawngroup.id` |
| `zone` | Zone short name |
| `version` | Zone version (0 = default) |
| `x`, `y`, `z`, `heading` | World coordinates and facing |
| `respawntime` | Seconds between spawns |
| `variance` | Random variance added to respawntime |
| `pathgrid` | FK to `grid.id` -- patrol path for this spawn |
| `path_when_zone_idle` | Whether to path when zone is idle |
| `_condition`, `cond_value` | Conditional spawning |
| `animation` | Spawn animation state |
| `min_expansion`, `max_expansion` | Expansion filtering |
| `content_flags`, `content_flags_disabled` | Feature flag filtering |

### spawngroup (spawn pool)

Groups one or more NPCs that share a spawn point. When the spawn timer fires, one NPC is chosen from the group.

| Column | Purpose |
|---|---|
| `id` | Unique spawngroup ID |
| `name` | Descriptive name |
| `spawn_limit` | Max simultaneous spawns |
| `dist`, `max_x`, `min_x`, `max_y`, `min_y` | Random spawn offset area |
| `delay`, `mindelay` | Additional delay parameters |
| `despawn` | Despawn behavior flag |
| `despawn_timer` | Despawn timer in ms |
| `wp_spawns` | Whether to spawn at waypoints |

### spawnentry (NPC selection)

Maps which NPCs can spawn in a given spawngroup, with probability.

| Column | Purpose |
|---|---|
| `spawngroupID` | FK to `spawngroup.id` |
| `npcID` | FK to `npc_types.id` |
| `chance` | Spawn probability (0-100, all entries should sum to 100) |
| `condition_value_filter` | Conditional filtering |
| `min_time`, `max_time` | Time-of-day restrictions |
| `min_expansion`, `max_expansion` | Expansion filtering |

### grid / grid_entries (patrol paths)

| Table | Key Columns |
|---|---|
| `grid` | `id`, `zoneid`, `type` (wander behavior), `type2` (pause behavior) |
| `grid_entries` | `gridid`, `zoneid`, `number` (waypoint order), `x/y/z/heading`, `pause` (seconds), `centerpoint` |

Grid `type` values: 0=stop, 1=loop, 2=depop, 3=random, 4=halt, 5=random halt, 6=patrol, 7=random-centerpoint, 8=random-patrol, 9=random-patrol-halt.

**Example query -- find all NPCs that can spawn in Crushbone:**
```sql
SELECT se.npcID, nt.name, se.chance, s2.x, s2.y, s2.z
FROM spawn2 s2
JOIN spawnentry se ON s2.spawngroupID = se.spawngroupID
JOIN npc_types nt ON se.npcID = nt.id
WHERE s2.zone = 'crushbone'
ORDER BY s2.id;
```

## Loot System

A four-table chain determines what drops from an NPC:

```
npc_types.loottable_id --> loottable --> loottable_entries --> lootdrop --> lootdrop_entries --> items
```

### loottable (loot table header)

| Column | Purpose |
|---|---|
| `id` | Unique ID (referenced by `npc_types.loottable_id`) |
| `name` | Descriptive name |
| `mincash`, `maxcash` | Copper range for cash drop |
| `avgcoin` | Average coin (used by some display tools) |
| `done` | Loot-editing completion flag |

### loottable_entries (links loottable to lootdrops)

| Column | Purpose |
|---|---|
| `loottable_id` | FK to `loottable.id` |
| `lootdrop_id` | FK to `lootdrop.id` |
| `multiplier` | How many times to roll this lootdrop |
| `droplimit` | Max items from this lootdrop (0 = unlimited) |
| `mindrop` | Minimum items that must drop |
| `probability` | Probability this lootdrop is rolled at all (0-100) |

### lootdrop (loot pool header)

| Column | Purpose |
|---|---|
| `id` | Unique ID |
| `name` | Descriptive name |

### lootdrop_entries (individual items)

| Column | Purpose |
|---|---|
| `lootdrop_id` | FK to `lootdrop.id` |
| `item_id` | FK to `items.id` |
| `item_charges` | Number of charges (for charged items) |
| `equip_item` | Whether to visually equip this item on the NPC |
| `chance` | Drop probability (0-100) |
| `disabled_chance` | Chance when content is disabled |
| `trivial_min_level`, `trivial_max_level` | Trivial loot filtering by player level |
| `multiplier` | How many of this item can drop |
| `npc_min_level`, `npc_max_level` | Only drop from NPCs in this level range |

### global_loot

Applied automatically to NPCs matching certain criteria (level range, race, class, bodytype, zone, rare/raid flags). References `loottable_id`.

**Example query -- what can a specific NPC drop:**
```sql
SELECT i.Name, i.id, lde.chance
FROM npc_types nt
JOIN loottable lt ON nt.loottable_id = lt.id
JOIN loottable_entries lte ON lt.id = lte.loottable_id
JOIN lootdrop ld ON lte.lootdrop_id = ld.id
JOIN lootdrop_entries lde ON ld.id = lde.lootdrop_id
JOIN items i ON lde.item_id = i.id
WHERE nt.id = 58081
ORDER BY lde.chance DESC;
```

## Item System

### items (~900 columns)

The `items` table is the largest by column count. It contains every item property from the original EQ client data files. Key column groups:

**Identity**: `id`, `Name`, `lore`, `idfile` (model), `itemtype`

**Stats**: `aagi/acha/adex/aint/asta/astr/awis` (stat bonuses), `ac`, `hp`, `mana`, `endur`, `attack`, `accuracy`, `avoidance`, `haste`

**Damage**: `damage`, `delay`, `backstabdmg`, `banedmgamt`, `banedmgrace`, `banedmgbody`, `elemental_dmg_amt/type`

**Resists**: `mr/cr/dr/fr/pr/svcorruption`

**Restrictions**: `classes` (bitmask), `races` (bitmask), `deity` (bitmask), `slots` (bitmask), `reqlevel`, `reclevel`, `reqskill`

**Container**: `bagsize`, `bagslots`, `bagtype`, `bagwr` (weight reduction)

**Spell effects**: `clickeffect/clicktype/clicklevel`, `proceffect/proctype/proclevel`, `worneffect/worntype/wornlevel`, `focuseffect/focustype/focuslevel`, `scrolleffect/scrolltype/scrolllevel`, `bardeffect/bardtype/bardlevel`

**Augmentation**: `augtype`, `augslot1type` through `augslot6type`, `augrestrict`

**Economics**: `price`, `sellrate`, `nodrop`, `norent`, `notransfer`, `loregroup`

## Spell System

### spells_new (~760 columns)

Massive table mirroring the spell data files. Each spell has 12 effect slots with parallel column groups:

- `effectid1-12` -- spell effect type
- `effect_base_value1-12` -- base value
- `effect_limit_value1-12` -- limit value
- `max1-12` -- max value
- `formula1-12` -- scaling formula

**Other key columns**: `name`, `mana`, `cast_time`, `recovery_time`, `recast_time`, `range_`, `aoerange`, `buffduration`, `buffdurationformula`, `targettype`, `resisttype`, `classes1-16` (min level per class), `skill`, `spell_group`, `deities0-16`.

**Message strings**: `you_cast`, `other_casts`, `cast_on_you`, `cast_on_other`, `spell_fades`

### NPC spell lists

```
npc_types.npc_spells_id --> npc_spells --> npc_spells_entries --> spells_new
```

`npc_spells` defines casting behavior (proc chances, recast delays for engaged/pursuing/idle states, buff/debuff/damage cast probabilities). Supports inheritance via `parent_list` (a child list merges with its parent).

`npc_spells_entries` maps individual spells with level ranges, priority, and HP thresholds (e.g., heal when below 50% HP).

## Zone System

### zone (~100 columns)

Defines every zone in the game world.

**Identity**: `id`, `zoneidnumber`, `short_name`, `long_name`, `version`

**Safe point**: `safe_x/y/z/heading` -- where players land after death or /rewind

**Environment**: `sky`, `fog_*` (4 fog layers with colors and clip planes), `rain_chance1-4`, `snow_chance1-4`, `gravity`, `underworld` (death plane Z), `lava_damage`

**Gameplay**: `zone_exp_multiplier`, `min_level`, `max_level`, `maxclients`, `canbind`, `cancombat`, `canlevitate`, `castoutdoor`, `hotzone`

**Rendering**: `minclip`, `maxclip`, `ztype` (indoor/outdoor), `walkspeed`

**Server**: `ruleset` (FK to `rule_sets.ruleset_id`), `idle_when_empty`, `seconds_before_idle`, `shutdowndelay`

### zone_points

Zone connections. Each entry defines a trigger region in one zone that teleports to a target location in another zone. Key columns: `zone`, `number`, `x/y/z` (trigger), `target_zone_id`, `target_x/y/z` (destination), `buffer` (trigger radius), `is_virtual` (non-physical zone line), `height/width` (trigger dimensions).

### doors

Zone geometry that interacts with players. Includes actual doors, zone portals, switches, teleporter pads, and decorative objects. `dest_zone/dest_x/y/z` columns define teleportation targets. `keyitem` references `items.id` for locked doors.

### object

World-placed interactable objects (tradeskill containers, barrels, etc.). `type` determines behavior. `itemid` references `items.id` for clickable items.

## Player Data

### character_data (~128 columns)

The master character record containing all persistent character state not broken out into sub-tables.

**Identity**: `id`, `account_id`, `name`, `last_name`, `title`, `suffix`

**Position**: `zone_id`, `zone_instance`, `x/y/z/heading`

**Core stats**: `level`, `race`, `class_`, `gender`, `deity`, `exp`, `aa_points`, `aa_exp`, `aa_points_spent`

**Attributes**: `str/sta/cha/dex/int_/agi/wis`

**Resources**: `cur_hp`, `mana`, `endurance`, `hunger_level`, `thirst_level`, `intoxication`

**Currency** is in a separate `character_currency` table, not in `character_data`.

**Timestamps**: `birthday`, `last_login`, `time_played`, `first_login`

**Flags**: `gm`, `anon`, `pvp_status`, `lfp`, `lfg`, `exp_enabled`, `autosplit_enabled`, `ingame`

**Soft delete**: `deleted_at` -- characters are soft-deleted, not removed

### account

Server-side account data: `name`, `password`, `status` (GM level 0-255), `sharedplat`, `ls_id` (login server ID), GM flags (`gmspeed`, `invulnerable`, `flymode`, `hideme`), ban/suspend info.

## Faction System

Factions control NPC hostility toward players. The system involves several interconnected tables:

```
npc_types.npc_faction_id --> npc_faction --> npc_faction_entries --> faction_list
                                                                     faction_list_mod
                                                                     faction_base_data
                             faction_values (per-character standings)
                             faction_association (cross-faction relationships)
```

### faction_list

Master list of all factions: `id`, `name`, `base` (default standing).

### faction_list_mod

Starting faction adjustments based on race, class, or deity. `mod_name` is one of the race/class/deity names; `mod` is the adjustment value. So different races start with different faction standings.

### faction_base_data

Min/max clamp values for each faction. Prevents faction from going above or below certain thresholds.

### npc_faction

Header for an NPC's faction configuration: `primaryfaction` (FK to `faction_list.id`), `ignore_primary_assist`.

### npc_faction_entries

Defines how killing an NPC with this faction set affects player faction standings: `faction_id`, `value` (amount of faction hit), `npc_value` (the NPC's personal attitude override: -1=hostile, 0=use faction, 1=friendly), `temp` (whether the hit is temporary).

### faction_values

Runtime table storing each character's current standing with each faction: `char_id`, `faction_id`, `current_value`.

### faction_association

Faction-to-faction relationships. When a player's standing with one faction changes, associated factions may also shift. Contains min/max modifiers for up to 18 associated factions per entry.

## Task / Quest System

### tasks

Master task definitions: `type` (0=Task, 1=Shared Task, 2=Quest), `title`, `description`, `duration`, rewards (`cash_reward`, `exp_reward`, `reward_id_list`, `faction_reward`), level restrictions, player count restrictions, replay timers.

### task_activities

The steps within a task: `activitytype` (kill, deliver, explore, loot, speak_with, tradeskill, fish, forage, etc.), `target_name`, `goalcount`, `zones`, `npc_match_list`, `item_id_list`, coordinate bounds for explore activities.

### character_tasks / character_activities

Per-character task and activity progress tracking.

## Tradeskill System

### tradeskill_recipe

Recipe definitions: `tradeskill` (skill ID: 59=Baking, 60=Tailoring, etc.), `skillneeded`, `trivial` (trivial level), `nofail`, `must_learn`, `learned_by_item_id`.

### tradeskill_recipe_entries

Components and products for each recipe: maps `recipe_id` to `item_id` with `successcount`, `failcount`, `componentcount`, `iscontainer`, `isnew` (product vs component flag).

## Merchant System

### merchantlist

Links merchant NPCs to their item inventories. `merchantid` is referenced by `npc_types.merchant_id`. Each row is one item slot: `item` (FK to `items.id`), `slot` (display order), `faction_required`, `level_required`, `classes_required` (bitmask), `probability` (100 = always available), `alt_currency_cost`.

Supports data bucket conditions: `bucket_name`, `bucket_value`, `bucket_comparison` for dynamic availability.

## Alternate Advancement (AA) System

```
aa_ability --> aa_ranks --> aa_rank_effects
                        --> aa_rank_prereqs
```

`aa_ability`: top-level AA (name, class bitmask, race bitmask, deity filter, `first_rank_id`).

`aa_ranks`: each purchasable rank (cost in AA points, spell effect, next rank pointer, expansion, etc.).

`aa_rank_effects`: spell effects granted by each rank.

## Bot System

Bots are player-created AI companions. The `bot_data` table mirrors many `character_data` columns:

`bot_id`, `owner_id` (FK to `character_data.id`), `spells_id` (FK to `npc_spells.id`), `name`, `race`, `class`, `level`, `deity`, appearance columns, stats (`ac/atk/hp/mana/str/sta/cha/dex/int_/agi/wis`), resists.

Supporting tables parallel the `character_*` pattern: `bot_buffs`, `bot_inventories`, `bot_pets`, `bot_timers`, etc.

Key difference from mercs: bots are fully player-customizable (name, race/class combo, equipment). They are **not** limited to a few templates.

---

## Mercenary System

Mercenaries are template-based hirelings available from NPC merchants. The system is more rigid than bots:

```
merc_merchant_entries --> merc_merchant_templates --> merc_merchant_template_entries --> merc_templates
merc_templates --> merc_types, merc_subtypes, merc_npc_types
merc_templates --> merc_stats (by level)
merc_templates --> merc_spell_lists --> merc_spell_list_entries
merc_templates --> merc_armorinfo, merc_weaponinfo
merc_templates --> merc_stance_entries
```

Active mercs are stored in `mercs` table with `OwnerCharacterID` and `TemplateID`.

## Rule System

### rule_sets / rule_values

The server's behavior is configurable via named rules. `rule_sets` defines rulesets (typically just ruleset 1). `rule_values` stores key-value pairs:

- `ruleset_id` -- which ruleset
- `rule_name` -- hierarchical name like `"Mercs:ResurrectRadius"` or `"Character:MaxLevel"`
- `rule_value` -- string value
- `notes` -- human-readable description

Rules are loaded at server boot and cached. A zone can override the default ruleset via `zone.ruleset`.

Common rule categories: `Character`, `Combat`, `Spells`, `Bots`, `Mercs`, `World`, `Zone`, `Guild`, `AA`, `TaskSystem`, `Adventure`, `NPC`, `Pets`.

## Content Flags and Expansion Filtering

### content_flags

Feature flags that can enable/disable swaths of content: `flag_name`, `enabled`. Many content tables have `content_flags` and `content_flags_disabled` columns that reference these flag names.

### Expansion Filtering

Most content tables include `min_expansion` and `max_expansion` columns (int8). These restrict content to specific expansion ranges. For the EQ project (Classic through Luclin), you would set expansion limits to include only expansions -1 through 3 (or whatever the configured range is).

The expansion values use a specific encoding:
- -1 = All expansions
- 0 = Classic
- 1 = Kunark
- 2 = Velious
- 3 = Luclin
- 4+ = later expansions

This filtering is applied at load time by the C++ repository queries.

## Data Buckets (Dynamic Key-Value Store)

### data_buckets

A flexible key-value store used by quests and server logic. Scoped by multiple dimensions:

| Column | Purpose |
|---|---|
| `key_` | The data key name |
| `value` | The stored value (string) |
| `expires` | Unix timestamp expiry (0 = never) |
| `account_id` | Scope to account (0 = not scoped) |
| `character_id` | Scope to character (0 = not scoped) |
| `npc_id` | Scope to NPC type (0 = not scoped) |
| `bot_id` | Scope to bot (0 = not scoped) |
| `zone_id` | Scope to zone (0 = not scoped) |
| `instance_id` | Scope to instance (0 = not scoped) |

Data buckets replace the older `quest_globals` system and are the recommended way to persist quest state. They can also be used by `merchantlist` conditions for dynamic merchant availability.

## Schema Migrations

The schema migration system is embedded in the C++ server binary, not in standalone SQL files.

### Migration Manifests

| File | Purpose |
|---|---|
| `common/database/database_update_manifest.h` | Main schema migrations (versions 9000+) |
| `common/database/database_update_manifest_bots.h` | Bot-specific migrations (versions 9035+) |
| `common/database/database_update_manifest_custom.h` | Custom server migrations |

### How It Works

Each `ManifestEntry` contains:
- `version` -- sequential version number
- `description` -- migration name (usually date-based)
- `check` -- SQL query to test if migration is needed
- `condition` -- how to evaluate the check result (`empty`, `not_empty`, `contains`, `match`, `missing`)
- `sql` -- the DDL/DML to execute

At server boot, `DatabaseUpdate::CheckDbUpdates()` reads the current version from `db_version` table, compares against `CURRENT_BINARY_DATABASE_VERSION` (9328 as of this writing), and runs any pending migrations.

### Version Tracking

The `db_version` table has three columns:
- `version` -- main schema version
- `bots_version` -- bot subsystem version
- `custom_version` -- custom server version

## Stored Procedures and Triggers

The PEQ database traditionally does **not** use stored procedures or triggers. All business logic lives in the C++ server code. This is a deliberate design choice -- the database is treated as a dumb data store, with the application layer handling all logic.

To verify: `SHOW PROCEDURE STATUS WHERE Db='peq'` and `SHOW TRIGGERS FROM peq` should return empty results.

## Where to Start

### For someone wanting to modify game content:

1. **Add an NPC**: Insert into `npc_types`, create a `spawngroup`, add a `spawnentry` linking them, create a `spawn2` point in the target zone. Optionally create a `grid`/`grid_entries` for patrol path.

2. **Add loot to an NPC**: Create a `lootdrop` and `lootdrop_entries` (referencing `items`), create a `loottable` and `loottable_entries` (referencing the lootdrop), set `npc_types.loottable_id`.

3. **Add a merchant**: Insert rows into `merchantlist` with the desired `merchantid`, then set `npc_types.merchant_id` on the NPC.

4. **Create a quest task**: Insert into `tasks` and `task_activities`.

5. **Adjust server behavior**: Modify `rule_values` entries.

6. **Control expansion content**: Set `min_expansion`/`max_expansion` on content rows, or use `content_flags`.

### Key tools for content editing:

- **Spire** (web UI at port 3000) -- visual editors for most content tables
- **Direct SQL** via `docker exec akk-stack-mariadb-1 mysql -ueqemu -p'...' peq`
- **#reloadworld, #reloadzone** in-game commands to hot-reload content changes

## Companion Recruitment System Relevance

The project's signature feature -- recruiting any NPC as a companion -- will need to interface with several table groups:

### Tables to READ (NPC capabilities)

| Table | Why |
|---|---|
| `npc_types` | Source of all NPC stats, abilities, appearance -- the "template" for companions |
| `npc_spells` / `npc_spells_entries` | What spells the NPC can cast (determines companion spell behavior) |
| `npc_faction` / `npc_faction_entries` | NPC faction relationships (affects recruitment eligibility, companion faction reactions) |
| `npc_scale_global_base` | Scaling data for auto-leveled NPCs |
| `spells_new` | Spell details for companion abilities |

### Tables to WRITE (companion state)

The bot system is the closest existing analog. `bot_data` and its satellite tables (`bot_buffs`, `bot_inventories`, `bot_pets`, etc.) demonstrate how to persist a companion:

| Table Pattern | Relevance |
|---|---|
| `bot_data` | Model for companion persistence (owner, stats, appearance) |
| `bot_inventories` | Model for companion equipment |
| `bot_buffs` | Model for companion buff state |
| `bot_spells_entries` | Model for companion spell lists |
| `bot_settings` / `bot_owner_options` | Model for per-owner customization |
| `bot_heal_rotations*` | Model for AI behavior configuration |

### New tables likely needed

The companion system will probably need custom tables such as:

- `companion_data` -- recruited NPC instances (owner_id, source_npc_type_id, current stats, recruitment timestamp, loyalty, etc.)
- `companion_*` satellite tables following the bot pattern (buffs, inventory, timers, etc.)
- `companion_recruitment_rules` -- configurable rules for which NPCs can be recruited
- Possibly extensions to `npc_types` (or a mapping table) for recruitment eligibility flags

### Key insight

The bot system already solves the "NPC as party member" persistence problem. The companion recruitment system adds a new dimension: **the NPC's identity comes from `npc_types` rather than being player-created from scratch**. This means the companion system needs to bridge between the content tables (read-only NPC definitions) and the operational tables (mutable companion state).

The `data_buckets` table is also relevant for storing recruitment quest state and companion progression data without needing new table schemas immediately.
